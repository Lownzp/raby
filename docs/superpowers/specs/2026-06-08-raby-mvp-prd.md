## 1. MVP 定义

Raby 的 MVP 是一个 Android 本地优先 App,让个人养兔用户可以完成三件核心事:

1. 建立兔子基础档案
2. 持续记录生活日记、照片和体重
3. 通过图表、热力图和大事记回看兔子的成长状态

MVP 不追求完整宠物医疗管理,也不追求社交、云同步或家庭协作。第一目标是让用户愿意长期打开 App 记录,并且相信自己的数据不会轻易丢失。

---

## 2. 目标用户与核心场景

### 2.1 目标用户

- 家里养 1 到 5 只宠物兔的个人用户
- 默认主推单只兔子,但允许后续添加多只
- 对体重变化、日常饮食和成长回忆有记录需求
- 能接受本地存储和手动备份

### 2.2 核心场景

| 场景 | 用户目标 | MVP 响应 |
|---|---|---|
| 初次使用 | 录入兔子的名字、生日、性别、品种等基础信息 | 提供轻量建档流程,没有档案时优先引导创建 |
| 日常记录 | 写一条日记,配几张照片,打上标签 | 提供时间轴、发布入口、多图选择和标签 |
| 体重记录 | 输入今天的体重,后续观察趋势 | 提供体重录入、历史列表、折线图和异常提示 |
| 成长回看 | 找到重要时刻,例如第一次剪指甲 | 用里程碑标签汇总到兔生大事记 |
| 数据备份 | 换手机或怕数据丢失 | 支持导出 ZIP,至少 v1.0 前完成可还原导入 |

---

## 3. 成功标准

MVP 达标需要满足以下结果:

- 用户可以在无网络环境下完成建档、写日记、上传照片、记录体重和查看图表。
- 用户首次打开 App 后,5 分钟内可以完成一只兔子的建档和第一条记录。
- 体重记录能稳定生成趋势图,并能展示周、月、年、全部四种时间范围。
- 日记时间轴支持按时间倒序浏览,照片缩略图能正常加载,删除和编辑不会遗留不可见脏数据。
- 导出数据包包含数据库、媒体文件和元数据,导入后核心数据可完整恢复。
- App 不依赖云服务即可运行,本地数据结构为未来云同步保留 UUID、软删除和更新时间字段。

## 4. 范围边界

### 4.1 MVP 必须做

| 模块 | 必须能力 | 说明 |
|---|---|---|
| 基础档案 | 新建、编辑、查看兔子档案 | 单只优先,多只作为 v1.0 能力 |
| 日记相册 | 文本、照片、标签、时间轴 | v0.1 先支持照片,视频后置 |
| 体重管理 | 体重录入、列表、图表、备注 | 单位固定为 g |
| 数据分析 | 记录热力图、兔生大事记、体重异常提示 | 先做本地计算,不做复杂统计 |
| 数据备份 | ZIP 导出、ZIP 导入 | 导出优先,导入在 v0.2 到 v1.0 完成 |
| 设置 | 备份入口、提醒设置、基础偏好 | 保持轻量 |

### 4.2 MVP 可以延后

| 能力 | 延后到 | 原因 |
|---|---|---|
| 短视频记录 | v0.2 或 v1.0 | 压缩、预览和失败兜底成本高 |
| 那年今日通知 | v0.2 | 需要先有足够日记数据 |
| 多兔切换完整体验 | v1.0 | 数据模型先支持,入口体验后补 |
| 导入合并策略 | v1.0 | 只覆盖容易,合并需要冲突处理 |
| BCS 体况评分 | v0.2 | 价值明确,但不是首版输入必需 |
| 自定义体重提醒 | v0.2 | 通知权限和关闭逻辑需要打磨 |

### 4.3 明确不做

- 云同步、账号系统、登录注册
- 社交 feed、关注、点赞、评论
- GPS、地点、路线
- 繁殖、配种、血统管理
- 家庭成员协作
- 完整健康管理,包括疫苗、驱虫、就诊、用药流程
- 标签云、活跃度趋势、照片占用统计

## 5. 版本分期

### 5.1 v0.1:可记录内测版

目标:用户可以真正开始记录一只兔子的档案、日记和体重。

必须包含:

- Flutter Android 工程骨架
- 基础主题、路由、Riverpod 状态管理
- Drift 数据库初始化
- 兔子档案新建、编辑、查看
- 日记新建、编辑、删除、时间轴列表
- 日记支持文本和最多 9 张照片
- 标签基础能力,包含系统标签和自定义标签
- 体重记录新建、编辑、删除、历史列表
- 体重折线图,支持全部数据视图
- 设置页基础入口

v0.1 不要求:

- 视频
- 导入导出
- 本地通知
- 多兔完整切换
- 热力图
- 异常预警

验收标准:

- 新用户从空数据启动,可以创建一只兔子并写一条带照片的日记。
- 用户可以录入至少 10 条体重记录,图表能正确展示趋势。
- 退出 App 再进入后,档案、日记、照片和体重数据仍然存在。
- 删除日记后,时间轴不再展示该记录。

### 5.2 v0.2:备份与回看版

目标:补齐长期使用需要的回看、提醒和备份能力。

必须包含:

- 记录热力图
- 兔生大事记,汇总带里程碑标签的日记
- 体重图表支持周、月、年、全部切换
- 体重异常提示,默认阈值为 7 天内变化超过 10%
- BCS 体况评分 1 到 5
- 自定义体重提醒
- 那年今日本地提醒
- ZIP 数据导出
- 短视频记录初版,单条上限 30 秒,压缩失败时保留原文件

验收标准:

- 用户至少有 30 天记录时,热力图能正确显示有记录日期。
- 给日记添加里程碑标签后,兔生大事记能自动出现对应条目。
- 导出的 ZIP 包包含数据库、媒体文件和 manifest 元数据。
- 体重提醒关闭后不会继续发通知。

### 5.3 v1.0:完整 MVP 发布版

目标:达到可长期自用和小范围分发的稳定版本。

必须包含:

- 多兔档案管理和切换
- 导入 ZIP,支持覆盖模式
- 导入 ZIP,支持基础合并模式
- 媒体文件丢失、导入失败、压缩失败的错误提示
- 软删除和更新时间字段完整落库
- 数据库迁移策略
- 基础异常日志
- APK 发布配置
- 核心单元测试、Widget 测试和关键流程集成测试

验收标准:

- 从 v0.1 或 v0.2 数据升级到 v1.0 后,已有数据不丢失。
- 导出再导入到空 App 后,档案、日记、照片、体重和标签能恢复。
- 多兔数据不会串档,不同兔子的日记和体重记录相互隔离。
- 无网络状态下所有核心功能可用。

## 7. 功能需求

### 7.1 基础档案

字段:

| 字段 | v0.1 | 规则 |
|---|---|---|
| 名字 | 必填 | 1 到 20 个字符 |
| 性别 | 必填 | 公、母、未知 |
| 生日或领养日 | 必填 | 至少填一个日期 |
| 品种 | 必填 | 可从预置项选择,也可自定义 |
| 毛色 | 必填 | 文本或预置选项 |
| 头像 | 可选 | 从相册选择或拍照 |
| 绝育状态 | 可选 | 未绝育、已绝育、未知 |
| 绝育日期 | 可选 | 仅已绝育时展示 |
| 初始体重 | 可选 | 单位 g |
| 性格标签 | 可选 | 多选 |
| 喜好 | 可选 | 食物、玩具自由文本 |
| 离世日期 | 可选 | v1.0 前保留字段 |

规则:

- 没有任何兔子档案时,启动后优先进入建档流程。
- v0.1 只要求默认兔子可用。
- 所有兔子数据用 UUID。
- 删除兔子需要二次确认。v1.0 前优先软删除。

### 7.2 日记相册

单条日记字段:

- 兔子 ID
- 记录日期时间
- 正文
- 照片列表
- 视频列表
- 标签列表
- 是否里程碑
- 创建时间、更新时间、删除时间

v0.1 规则:

- 正文可以为空,但照片和正文至少有一个。
- 照片单条最多 9 张。
- 时间默认当前时间,允许用户修改。
- 时间轴按记录时间倒序展示。
- 支持编辑和删除。
- 支持系统标签和自定义标签。

v0.2 规则:

- 支持短视频,单条上限 30 秒。
- 支持里程碑模板,本质是带特殊标签的日记。
- 正文超过 3 行时默认折叠。

验收标准:

- 连续创建 50 条日记后,时间轴滚动不卡顿到影响使用。
- 日记编辑后,更新时间更新,时间轴位置按记录时间重新排序。
- 删除日记后,对应媒体不再被业务引用。

### 7.3 体重管理

字段:

- 兔子 ID
- 记录日期
- 体重 g
- 备注
- 照片
- BCS 体况评分
- 创建时间、更新时间、删除时间

规则:

- 体重必须大于 0。
- 同一天允许多条记录,不强行覆盖。
- 图表默认展示最近数据,可切换周、月、年、全部。
- 异常预警默认规则为 7 天内变化超过 10%。
- 异常预警只提示,不做医疗判断。

验收标准:

- 输入 900g、1000g、1100g 等数据时,图表纵轴和单位展示正确。
- 修改记录日期后,列表和图表都按新日期更新。
- 异常提示文案避免诊断语气,只提醒用户关注。

### 7.4 数据分析

MVP 分析只围绕记录习惯和成长回看:

| 分析项 | 版本 | 规则 |
|---|---|---|
| 体重折线图 | v0.1 | 必须 |
| 记录热力图 | v0.2 | 统计每天是否有日记或体重记录 |
| 兔生大事记 | v0.2 | 汇总里程碑日记和健康事件 |
| 体重异常提示 | v0.2 | 基于最近记录本地计算 |
| 那年今日 | v0.2 | 本地通知和入口提示 |

不做:

- 标签云
- 活跃度趋势
- 照片占用统计
- AI 健康判断

### 7.5 数据导入导出

导出目标:

- 用户可以通过系统分享菜单保存 ZIP。
- ZIP 至少包含数据库、媒体文件和 manifest。
- manifest 记录 App 数据版本、导出时间、文件清单。

v0.2 先做导出。

v1.0 必须支持导入:

- 覆盖模式:清空当前数据后还原 ZIP。
- 合并模式:按 UUID 合并,更新时间较新的数据优先。
- 媒体文件按路径或 hash 复用,避免重复复制。

验收标准:

- 导出 ZIP 后,解压可看到 manifest 和媒体目录。
- 导入失败不会清空当前已有数据。
- 导入过程中有进度和错误提示。

## 8. 数据与架构约束

MVP 必须遵守以下约束:

- 所有业务实体使用 UUID v4。
- 所有主要表保留 `createdAt`、`updatedAt`、`deletedAt`。
- 删除默认软删除,导出时可包含软删除记录。
- Domain 层不依赖 Flutter、Drift、Riverpod。
- Repository 接口放在 Domain,实现放在 Data。
- 媒体文件不直接存数据库,数据库只存元数据和相对路径。
- 数据库迁移从 v0.1 开始保留版本号。

## 10. 风险与取舍

| 风险 | 影响 | MVP 取舍 |
|---|---|---|
| 视频压缩在部分 Android ROM 上失败 | 影响日记媒体体验 | 视频不进 v0.1,压缩失败保留原文件 |
| 导入合并复杂 | 可能导致数据覆盖错误 | v0.2 先导出,v1.0 再做导入合并 |
| 多兔体验增加页面复杂度 | 拖慢首版开发 | v0.1 单兔优先,v1.0 补完整切换 |
| 通知权限被系统限制 | 提醒可能不稳定 | 提醒作为辅助功能,核心记录不依赖通知 |
| 图表过度设计 | 影响开发节奏 | 只做体重趋势、热力图、大事记三类 |
