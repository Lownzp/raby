# Draft C：故事版

## 开场

想象一个用户第一次打开 Raby。他没有在找一个复杂的宠物医院系统，也不想注册账号、关注别人、刷社区。他只是想记住：今天兔子第一次跳上沙发，昨天体重变轻了一点，上个月剪了第一次指甲。

这些记录很小，但对用户很重要。Raby 的问题也从这里开始：如果这些内容会陪用户几年，我们应该从第一版就怎样设计它？

## 1. 第一幕：先让用户开始记录

Raby v0.1 的目标非常克制：让用户可以真正开始记录一只兔子。

不是云同步，不是社交，不是完整医疗档案。第一版只做：

- 建立兔子基础档案
- 写日记
- 添加照片
- 打标签
- 记录体重
- 看时间轴和体重趋势

这条闭环看似简单，却覆盖了一个长期记录 App 最重要的用户价值：我写下的东西，下次打开还能看到。

## 2. 第二幕：本地优先是一种承诺

Raby 选择本地优先，是因为它的核心数据更像个人日记，而不是社交内容。

用户可以在无网络状态下建档、写日记、选照片、记录体重。数据放在设备本地：

- 结构化数据在 SQLite
- 照片在 App 私有目录
- 数据库里只保存媒体相对路径

这让 Raby 在早期不需要账号系统，也不需要服务器。但它也要求本地数据结构不能随便设计。UUID、软删除、更新时间、相对路径，都是为了未来备份、导入、迁移和同步做准备。

## 3. 第三幕：页面不应该知道数据库

一个小 App 最容易犯的错，是在页面里越写越多：点击按钮时校验、复制图片、组装数据库 row、写表、跳转、报错都挤在一起。

Raby 把这些责任拆开：

- 页面负责输入和展示
- Controller 负责校验和命令
- Domain 负责业务对象
- Repository 负责领域接口
- Data 负责 Drift 和文件系统

于是写一条日记就不再是“页面写数据库”，而是一条清晰的路径：

```text
DiaryEditPage -> DiaryEditorController -> DiaryRepository -> Drift DAO -> SQLite
                         |
                         -> MediaStorageService -> App 私有目录
```

## 4. 第四幕：失败也要被设计

长期记录 App 不能只设计成功路径。用户选了 9 张照片，保存到一半失败怎么办？数据库写入失败但图片已经复制了怎么办？删除一条日记后，标签关系和媒体记录还展示怎么办？

Raby 在应用层处理这些情况：

- 单条日记最多 9 张照片
- 正文和照片至少有一个
- 媒体复制后再事务写库
- 写库失败时清理已复制文件
- 删除使用软删除，查询默认过滤

这些规则让用户看不到复杂性，但能感受到稳定。

## 5. 第五幕：为未来留门

Raby 的 v0.1 不做很多事：不做导入导出，不做多兔完整切换，不做热力图，不做提醒。

但它已经为这些能力留了门：

- 每条业务数据有 UUID
- 每张表有更新时间和软删除时间
- 每条日记属于某只兔
- 标签支持系统和自定义
- 媒体路径可被 ZIP 打包
- Repository 接口可以替换实现

这就是 MVP 架构的重点：不是现在就把未来都做完，而是不要把未来堵死。

## 结尾

Raby 是一个温柔的产品，但它背后的工程原则很硬：数据要可信，边界要清楚，失败要能恢复，未来要有余地。

如果用一句话总结：Raby 在做的不是一个“能写日记的 App”，而是一套可以长期保存生活记忆的本地优先系统。
