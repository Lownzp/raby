项目一句话

Raby 是一个本地优先的兔兔档案 Flutter App，帮助个人养兔用户在无网络环境下建立兔子档案、记录日记照片、管理标签、记录体重，并通过时间轴和图表回看成长状态。

它看起来是一个温柔的小产品，但工程上要守住三件事：数据要可信，边界要清楚，未来要有余地。

01 产品闭环

v0.1 的目标是可记录内测版。它不追求完整宠物医疗管理，也不追求社交、云同步或家庭协作，第一目标是让用户愿意长期打开 App 记录，并相信自己的数据不会轻易丢失。

02 架构分层

当前项目采用 feature-first 的轻量分层。它不是为了套模板，而是为了避免 UI、数据库和业务规则混在一起。

lib/
├── app/       路由、主题、全局 Provider
├── data/      Drift、SQLite、Repository 实现、媒体文件
├── domain/    实体、枚举、Repository 接口、业务异常
├── features/  页面、表单、Controller、局部状态
└── shared/    跨页面复用 UI
层职责
appMaterialApp.router、go_router、主题、Repository Provider、启动装配。
features/*/presentation页面布局、表单、列表、loading/empty/error/success 展示。
features/*/applicationRiverpod Provider、Controller、异步命令、输入到 Domain 的转换。
domain实体、枚举、Repository 接口、业务异常、纯业务规则。
dataDrift 表/DAO、Repository 实现、mapper、媒体文件服务。
推荐依赖方向 
app -> features -> domain
app -> data -> domain
features/application -> domain
data -> domain
shared -> app/theme
禁止穿透 
domain -> flutter / riverpod / drift
presentation -> data/database
presentation -> DAO / Table / Companion
data -> features

启动和依赖装配

Flutter 入口非常薄，应用状态由 Riverpod 的 ProviderScope 承载。

void main() {
  runApp(const ProviderScope(child: RabyApp()));
}

启动页先 seed 系统标签，再判断是否存在默认兔子。

final startupProvider = FutureProvider<StartupDestination>((ref) async {
  await ref.watch(tagRepositoryProvider).ensureSystemTagsSeeded();
  final rabbit = await ref.watch(rabbitRepositoryProvider).getDefaultRabbit();
  return rabbit == null
      ? StartupDestination.onboarding
      : StartupDestination.records;
});

Repository 装配集中在 app/providers，页面和 Controller 只依赖领域接口。

final appDatabaseProvider = Provider<AppDatabase>((ref) {
  final database = AppDatabase();
  ref.onDispose(database.close);
  return database;
});

final diaryRepositoryProvider = Provider<DiaryRepository>((ref) {
  return DriftDiaryRepository(
    ref.watch(appDatabaseProvider),
    uuid: ref.watch(uuidProvider),
  );
});

03 本地数据

Raby 使用 Drift 管理 SQLite。结构化数据进数据库，照片等媒体文件保存在 App 私有目录，数据库只保存相对路径。

rabbits

兔子基础档案，v0.1 展示默认兔，结构预留多兔。

diaries

日记主表，按兔和记录时间形成时间轴。

diary_media

日记图片与未来视频，保存相对路径与排序。

tags

系统标签和当前兔子的自定义标签。

diary_tags

日记和标签的多对多连接。

weight_records

体重列表和趋势图数据来源。

设计原则

  • 业务主表使用 UUID v4，不用自增 ID。
  • 保留 createdAt、updatedAt、deletedAt。
  • deletedAt == null 表示有效数据。
  • 创建、更新、删除时间使用 UTC epoch milliseconds。
  • 生日、领养日等纯日期使用 yyyy-MM-dd 文本。

media/rabbits/{rabbitId}/avatar/{mediaId}.jpg

04 关键流程

写日记

日记保存体现了 Raby 的边界：页面收集输入，Controller 执行业务命令，Repository 隐藏 Drift，Data 层处理事务。

DiaryEditPage
  -> DiaryEditorController
  -> MediaStorageService
  -> DiaryRepository
  -> Drift DAO transaction
  -> SQLite + App 私有媒体目录
void _validateContentAndMedia({
  required String? content,
  required List<DiaryMediaDraft> mediaDrafts,
}) {
  if (content == null && mediaDrafts.isEmpty) {
    throw const DomainValidationException('日记正文和照片至少需要一个');
  }
  if (mediaDrafts.length > 9) {
    throw const DomainValidationException('单条日记最多支持 9 张照片');
  }
}
  1. 生成 diaryId。
  2. 修剪正文并校验。
  3. 为每张新图片生成 mediaId。
  4. 复制图片到私有目录。
  5. 构造 Diary 和 DiaryMedia。
  6. 去重 tagIds。
  7. Repository 事务写入日记、媒体和标签连接。
  8. 失败时清理已经复制的媒体文件。

工程护栏

页面层

只处理布局、输入、反馈和导航。不直接创建复杂 Domain 对象图，不直接操作数据库或媒体文件。

应用层

处理 Provider、Controller、异步命令、UUID、Clock、跨字段业务校验和失败清理。

领域层

保持纯 Dart，只表达业务对象和能力，不 import Flutter、Riverpod、Drift。

数据层

负责 Drift row 与 Domain 的映射，不能依赖 features，不能把 row 泄露到 UI。
